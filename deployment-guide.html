How Deployment Works

Going from your laptop to repcommish.com — explained without the jargon

The whole idea: right now your app runs on your laptop — only you can see it. Deploying means making a copy that lives on a server somewhere in the world so anyone with the link can use it. That's it.

The restaurant analogy

Imagine your app is a restaurant.

  • Your laptop is the test kitchen. You cook, taste, adjust, throw things out, try again. No customers ever see it.
  • The server is the actual restaurant building. When you're happy with the dish, you bring the final version here, where customers can eat it.
  • The database is the warehouse next door. Both kitchens pull ingredients from the same warehouse, so the menu stays consistent.
  • Deploying is the truck that drives the finished recipe from your test kitchen to the restaurant.

The three pieces of your app

This is the most important thing to internalize. Your app isn't one thing — it's three things glued together. Each one gets deployed differently (or doesn't need to be deployed at all).

Piece 1

Frontend

The website itself — everything users click on, type into, look at. Written in React. Lives in the src/ folder.

Deploy: every time you change UI

Piece 2

Database

Where all the data lives — users, accounts, orders, payments. Lives in Supabase, which is already in the cloud 24/7. You don't really "deploy" the database, you change its structure when needed.

Deploy: only when you add tables/columns

Piece 3

Edge Functions

Tiny server-side scripts for things the browser can't do alone (sending emails, calling Claude AI, charging credit cards). Live in supabase/functions/.

Deploy: only when their code changes

Most of the time you'll only be deploying Piece 1 (the frontend). The other two stay put unless you specifically change them.

The frontend deploy — the one you'll do most

This is the bread and butter. Five steps. You can have Claude do all of these for you with one sentence.

Test it locally first

Run npm run dev, open localhost:5174, click around. Make sure the change actually works. Never skip this.

Commit to GitHub

Save your work to the repo so it's tracked. git add, git commit, git push.

Build a production version

npm run build takes your code and squishes it down into optimized files. Output lands in a folder called dist/.

Upload dist/ to the VPS

The "VPS" is the rented computer in the cloud that runs the live site. The files get copied there using scp (basically a secure version of drag-and-drop over the internet).

nginx serves the new files

nginx is the program on the VPS that hands files to anyone who visits repcommish.com. As soon as the new files land, nginx serves them. The site updates in ~30 seconds. No restart needed.

What to actually say to Claude: "Deploy the latest changes to repcommish.com." Claude will do all five steps. You don't need to memorize the commands.

The other two deploys (rarer)

Database changes

If you ask Claude to "add a column" or "create a new table," you've changed the database's structure. The change has to be applied to Supabase. Two ways this happens:

  • SQL Editor — go to supabase.com/dashboard, open your project, click "SQL Editor," paste the SQL Claude wrote, hit Run. Done.
  • schema.sql file — the project keeps a copy of the database structure at scripts/schema.sql. Update it so future setups stay in sync.
Be careful here. Database changes are permanent and affect all real users immediately. If you accidentally delete a column, the data in it is gone. Always have Claude show you the SQL first and explain what it does before you click Run.

Edge function changes

If you change anything inside supabase/functions/ — for example, the AI extraction logic, the Stripe webhook, the email sender — you have to redeploy that function to Supabase.

supabase functions deploy <function-name>

Claude does this for you when you say "deploy the updated edge function."

Local vs. production — the only two worlds

Local (your laptop)Production (live)
URL localhost:5174 repcommish.com & app.repcommish.com
Who sees it Only you Everyone on the internet
Database Real Supabase (same as production) Real Supabase (same as local)
When you see changes Instantly, as you save files After you deploy (~30 seconds)
Safe to break? Yes — you're the only one No — real users are using it
Important catch: Local and production share the same Supabase database. If you delete a user from your laptop, they're gone from production too. The "two worlds" are about the UI, not the data. To safely test risky data changes, you'd want a separate "staging" Supabase project — ask Claude to set one up if you ever need it.

Glossary — words Claude will use that sound scary

WordPlain English
DeployCopy your code from your laptop to the live server so real users can see it.
BuildCompress and optimize your code so it runs fast in a browser.
FrontendThe website itself — the stuff users see and click.
BackendEverything users can't see — the database, the logic, the integrations.
VPS"Virtual Private Server." A rented computer in the cloud that runs your site 24/7. Yours is at Hostinger.
nginxThe program on the VPS that hands your website's files to visitors. Pronounced "engine-x."
SupabaseThe company that hosts your database, user logins, and edge functions. You manage it at supabase.com/dashboard.
Edge functionA small piece of server-side code that runs in Supabase. Useful for anything the browser can't do alone (sending email, calling AI, etc.).
SSHSecure way to remotely control the VPS from your laptop. Like a private back-door key.
scp"Secure Copy." How files get sent from your laptop to the VPS. Like AirDrop but over the internet.
SSL / HTTPSThe padlock icon in browsers. Means traffic to your site is encrypted. Yours is set up with Let's Encrypt (free).
RLS"Row-Level Security." Supabase rules that control which users can see which rows of data. Critical for privacy.
MigrationA change to the database structure — like adding a column or renaming a table.
Rollback / revertUndo. Go back to the previous working version.

What can go wrong (and what to do)

  • You deploy and the site goes blank or shows an error. Tell Claude: "The site is broken — revert to the previous commit and redeploy." Done.
  • The local site works but the production site doesn't. Usually means a database column exists locally (in schema.sql) but you forgot to run the SQL on production Supabase. Tell Claude: "Compare schema.sql to what's actually in Supabase."
  • Edge function crashes. Check Supabase logs (supabase.com/dashboard → your project → Edge Functions → Logs). Tell Claude the error message and ask for a fix.
  • You're not sure if a change is already deployed. Visit the live site and try the feature. Or ask Claude: "Is the latest commit deployed to production?"
You almost never have to memorize commands. The trick to vibe-coding is knowing what to ask for. The glossary above is your translation dictionary — once you can name what's broken or what you want, Claude does the rest.
